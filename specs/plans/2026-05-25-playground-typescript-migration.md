# Plan: Migrate playground to TypeScript

## Goal

Convert the IronPLC playground (`playground/`) from JavaScript to TypeScript to
catch DOM/worker contract bugs at compile time, document the shapes flowing
between `app`, `worker.js`, and the WASM crate, and improve refactor safety as
new features (sparklines, dialect/allows flags, embed modes) accrete on top of
~840 LOC of untyped UI glue.

## Scope

In scope (converted to TypeScript):
- `playground/app.js` → `playground/src/app.ts`
- `playground/worker.js` → `playground/src/worker.ts`
- `playground/tests/e2e.spec.js` → `playground/tests/e2e.spec.ts`
- `playground/playwright.config.js` → `playground/playwright.config.ts`

Out of scope (kept as `.js`):
- `playground/posthog-init.js` — vendor snippet copied verbatim from PostHog
  docs. Stays as a pre-`<script>` shim so events fired during page load
  (`playground_loaded`) are queued. Already documented in the file header why
  rewriting it is undesirable.

The migration must not change runtime behavior. CI is the
`playground-e2e-test` Playwright job in `partial_playground_e2e.yaml`, invoked
via `just ci` in `playground/`.

## Architecture

The playground is plain ES modules served as static files — no bundler. To
keep that simple, this migration:

1. Sources live under `playground/src/` (browser code) and `playground/tests/`.
2. `tsc` emits browser-targeted ES modules straight into `_build/`. The
   compiled `_build/app.js` and `_build/worker.js` replace the previous
   hand-written copies one-for-one (same filenames the existing `index.html`
   references).
3. No bundler, no module resolution magic: `import` paths in `.ts` files use
   the same relative `.js` URLs the browser will see (TypeScript's
   `allowImportingTsExtensions: false` + emit-as-is means `import "./uPlot.esm.js"`
   compiles to the same string). This keeps the WASM and uPlot import URLs
   identical pre- and post-migration.
4. Playwright loads `*.spec.ts` natively via its bundled `tsx` loader (no
   `ts-node` config needed) — present in `@playwright/test ≥1.40`.
5. The Playwright config is also converted to TypeScript (`playwright.config.ts`).

### Type boundaries

Three external boundaries need type declarations:

- **WASM module** (`./pkg/ironplc_playground.js`): exports are generated by
  `wasm-pack --no-typescript`, so we hand-write a `.d.ts` ambient module
  covering `init`, `init_panic_hook`, `compile`, `run`, `run_source`,
  `load_program`, `step`, `reset_session`, `version`. The TS file lives next
  to the source code that consumes it but is not emitted.
- **uPlot ESM**: `./uPlot.esm.js` is shipped from `node_modules/uplot` and has
  no upstream `.d.ts`. We declare a minimal ambient module for the surface we
  actually use (`new uPlot(opts, data, target)`, `uPlot.paths.stepped`).
- **Window globals**: `clicky`, `posthog` (loaded by `posthog-init.js`) are
  declared on `Window` in a `globals.d.ts`.

### Worker message contracts

The `worker.ts ↔ app.ts` postMessage protocol becomes a discriminated union:

```ts
type WorkerRequest =
  | { id: number; command: "compile"; source: string; dialect?: string; allows?: string }
  | { id: number; command: "load_program"; source: string; cycleTimeUs: number; dialect?: string; allows?: string }
  | { id: number; command: "step"; scans: number }
  | { id: number; command: "reset" }
  | { id: number; command: "run"; bytecodeBase64: string; scans: number }
  | { id: number; command: "run_source"; source: string; scans: number; dialect?: string; allows?: string };

type WorkerResponse =
  | { type: "ready"; version: string }
  | { id?: number; type: "error"; error: string }
  | { id: number; type: "result"; json: string };
```

The JSON payload returned by the WASM crate is parsed in `app.ts` into:

```ts
type RunResult =
  | { ok: true; total_scans: number; variables: Variable[] }
  | { ok: false; diagnostics?: Diagnostic[]; error?: string };
```

`Variable` and `Diagnostic` are typed from the WASM crate's
`Variable`/`Diagnostic` serde structures.

- `playground/src/app.ts` (from `app.js`)
- `playground/src/worker.ts` (from `worker.js`)

- `playground/src/types/globals.d.ts` — `clicky`, `posthog` on `Window`.

- `playground/tests/e2e.spec.ts` (from `e2e.spec.js`)
- `playground/playwright.config.ts` (from `playwright.config.js`)

Deleted files (after migration):

## Verification

Inside `playground/`:

```
just compile   # builds wasm + runs tsc + assembles _build/
just test      # runs the full Playwright suite headlessly
just ci        # what CI runs
```

Acceptance:
- `_build/app.js` and `_build/worker.js` exist after `just compile`.
- All Playwright tests pass with no behavioral changes.
- `npx tsc --noEmit` reports zero errors.
- `index.html`, `style.css`, `posthog-init.js` are byte-identical to pre-migration.

## Non-goals

- No bundler (esbuild, Vite, Rollup). Static ES modules continue to work.
- No source maps in production (can be added later if debugging requires).
- No reorganization of UI logic — types describe what's there, no refactors.
- No conversion of `posthog-init.js` (intentional: see Scope).

## Risks

- **Playwright TS loading**: if the installed `@playwright/test` version does
  not auto-load `.ts` configs, fall back to keeping `playwright.config.js`
  and only converting the spec. Verify in commit 4 before deleting the `.js` config.
- **uPlot type surface drift**: the ambient declaration covers only what
  `app.ts` uses today (`new uPlot(opts, data, target)`, `uPlot.paths.stepped`).
  Adding new uPlot APIs later requires expanding `src/types/uplot.d.ts` — a
  one-line edit, not a rebuild.
- **WASM signature changes**: when the WASM crate's exports change,
  `src/types/wasm.d.ts` must be updated in lockstep. Same coupling exists
  today, just implicit; types make it explicit.
