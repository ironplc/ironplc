# SBOM in CI Implementation Plan

**Goal:** Generate a CycloneDX JSON Software Bill of Materials (SBOM) for each release, attach it to the GitHub Release as a sibling asset, AND embed it inside the Homebrew tarballs (Linux/macOS `.tar.gz`) and Windows NSIS installer so the SBOM travels with the binaries on the installed machine. Reviewers and security scanners (Grype, Trivy, Dependency-Track) can then audit transitive dependencies without re-resolving the dependency graph, and offline / air-gapped installs retain provenance — supporting US EO 14028 and EU Cyber Resilience Act compliance posture.

**Architecture:** Two SBOMs per release — one for the Rust workspace (covers all 19 crates and their transitive deps) and one for the VS Code extension (covers the npm dependency tree). Both are CycloneDX JSON spec version 1.5.

For the Rust workspace, `cargo-cyclonedx` is run with `--all` and emits one `bom.json` per crate; `cyclonedx-cli merge` then consolidates them into a single hierarchical workspace SBOM. For the VSIX, `@cyclonedx/cyclonedx-npm` reads `package-lock.json` directly via `--package-lock-only`.

The compiler SBOM is generated in a new `compiler-sbom` GitHub Actions job that runs once on `ubuntu-latest` before the 5-platform build matrix. The matrix jobs declare `needs: [compiler-sbom]`, download the SBOM as a build artifact, place it at `compiler/bom.cdx.json`, and the platform `_package-*` recipes embed it into the tarball / NSIS installer. A separate set of release-upload steps attaches the SBOM (under a versioned name) plus its `.sha256` sidecar to the GitHub Release.

Inside installers the SBOM uses a stable filename (`bom.cdx.json`); on the GitHub Release it uses a versioned filename (`ironplc-compiler-<version>.cdx.json`). `svenstaro/upload-release-action`'s `asset_name` parameter handles the rename.

## Verification

Local:
```bash
cd compiler && just sbom 0.215.0
jq '.bomFormat, .specVersion, (.components | length)' bom.cdx.json
# expect: "CycloneDX", "1.5", number in the hundreds
jq -r '.components[].name' bom.cdx.json | grep -E '^(ironplc-cli|ironplc-vm|ironplc-mcp)$'

cd ../integrations/vscode && just sbom 0.215.0
jq '.bomFormat, .specVersion, (.components | length)' bom.cdx.json
jq -r '.components[].name' bom.cdx.json | grep vscode-languageclient
```

## Out of Scope

- Embedding SBOM inside the VSIX (sibling release asset only).
- `cyclonedx validate` step in CI (cheap follow-up).
- Vulnerability scanning (Grype/Trivy) gated in CI (security follow-up).
- SPDX format alongside CycloneDX.
- Per-platform Rust SBOMs (workspace SBOM is platform-independent in practice).
- WASM playground SBOM.
